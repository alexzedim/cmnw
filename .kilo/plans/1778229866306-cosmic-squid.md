# Plan: Fix Realm Slug Resolution — Prevent `localeSlug` Leaking into `guid`

## Problem

When building character `guid` as `{name-slug}@{realm-slug}`, the `realm` value sometimes contains a `localeSlug` (e.g. `гордунни`) instead of the correct English `slug` (e.g. `gordunni`). This happens when the realm identifier comes from an external source (WCL API, Blizzard API, WoW Progress HTML, addon scan) and is passed through `toSlug()` (which is just `kebabCase()`) rather than being resolved against the `RealmsEntity.slug` column in the database.

**Key fact**: `RealmsEntity.slug` always holds the English EU slug (e.g. `gordunni`), never locale-specific variants like `гордунни`. The `localeSlug` column holds those. External APIs (WCL, Blizzard) may return either English or locale values — we must always resolve against the DB to get the canonical `slug`.

## Root Cause

`toSlug()` = `kebabCase()` which only transforms the string mechanically (lowercase, replace spaces with hyphens). It does **not** resolve "Гордунни" → "gordunni" because it has no knowledge of the canonical English slug stored in `RealmsEntity.slug`.

The solution is to **always resolve realm identifiers through the `RealmsCacheService`** (in-memory cache of all ~257 realms, already exists at `libs/resources/src/services/realms-cache.service.ts`) before forming the `guid`.

## Solution

### Step 1: Enhance `RealmsCacheService` — full coverage of all searchable fields

**File**: `libs/resources/src/services/realms-cache.service.ts`

**1a. Add `ticker` to the indexed maps:**

Currently missing from the cache. While unlikely to be a character realm input, it's a `varchar` column on `RealmsEntity` and should be indexed for completeness.

```typescript
private realmsByTicker = new Map<string, RealmsEntity>();

// In loadRealms(), add:
if (realm.ticker) {
  this.realmsByTicker.set(realm.ticker, realm);
  this.realmsByTicker.set(realm.ticker.toLowerCase(), realm);
}
```

Update `findRealm()` to also check `realmsByTicker`.

**1b. Add `resolveCanonicalSlug()` method:**

Thin wrapper that returns the canonical English `slug` for any input string:

```typescript
async resolveCanonicalSlug(query: string): Promise<string | null> {
  const realm = await this.findRealm(query);
  return realm?.slug ?? null;
}
```

**Current `findRealm()` search order** (already handles all external input types):
1. Exact match: `name` → `slug` → `localeName` → `localeSlug` → `ticker` (new)
2. `toSlug(query)` then check `slug` and `localeSlug` maps
3. Lowercase fallback on all maps

This covers: English names, English slugs, locale names (e.g. `Гордунни`), locale slugs (e.g. `гордунни`), tickers, and kebab-cased variants of all of the above.

### Step 2: Add `resolveRealmSlug()` static helper to `CharacterMessageDto`

**File**: `libs/resources/src/dto/queue/character-message.dto.ts`

Add a centralized static method that all call sites can use for consistent realm resolution:

```typescript
static async resolveRealmSlug(
  realmsCache: RealmsCacheService,
  realmInput: string,
): Promise<string> {
  const realm = await realmsCache.findRealm(realmInput);
  return realm?.slug ?? toSlug(realmInput);
}
```

This provides a fallback: if `findRealm()` returns `null`, fall back to `toSlug()` — same behavior as today.

## Risk Assessment

- **Low risk**: `RealmsCacheService` is already battle-tested in `api` and `market` services
- **Fallback**: If `findRealm()` returns `null` (realm not found), we fall back to `toSlug()` — same behavior as today, no worse
- **Performance**: In-memory cache (~460KB, ~257 realms), O(1) lookups, zero additional DB queries
- **Local caching**: Within batch methods (WCL, ladder), we cache resolved slugs per-call to avoid repeated lookups for the same realm
- **No DB changes**: Only code-level changes, no migrations needed
